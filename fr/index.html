Conventions de cette référence

Les mots en gras sont les plus importants, par exemple la commande ou le concept, et son utilisation. Dans le code d'exemple, nous mettons en valeur la commande utilisée : code_d_exemple.concept_a_montrer Dans les mêmes colonnes, les mots_italiques marquent les arguments/paramètres d'une commande/méthode.

Mais dans les descriptions ou le texte général, les mots italiques dénotent plutôt des concepts génériques, ou des concepts expliqués ailleurs (que ce soit dans cette page ou en général).

La console

La console (aussi appelée terminal, ligne de commande ou 'shell' est simplement un moyen d'interagir avec votre ordinateur. C'est comme une discussion avec lui : vous tapez une commande (un ordre ou une question), et il répond. Vous pouvez faire tout un tas de choses similaires à ce que vous feriez à l'aide d'une interface graphique. Comme c'est du texte, c'est plus facile de copier-coller les lignes ou de répéter et d'enchaîner des commandes. Grouper des commandes, souvent dans un fichier texte, pour les éxécuter en groupe s'appelle faire (ou lancer) un "script".

Selon votre OS (système d'exploitation), démarrer la console se fait de différentes manières.

  • Windows: ouvrez le menu démarrer et cherchez le terminal. Vous pouvez également choisir Exécuter... et taper cmd.
  • Mac: Ouvrez la recherche (Spotlight), tapez terminal, et lancez ce programme.
  • Linux: Le terminal devrait être une des principales options de votre menu principal, même s'ils diffèrent selon les distributions. S'il n'est pas mis en valeur, il est probablement dans un menu Utilitaires, ou nommé Terminal ou Console.

Bases de Ruby

Ruby on Rails est écrit avec le language de programmation Ruby. C'est pourquoi vous écrirez principalement votre code en Ruby. Il est donc important de comprendre les bases de ce language. Si vous souhaitez simplement essayer Ruby, tapez irb dans votre console. Vous pourrez ainsi experimenter avec le language. Pour quitter irb, taper exit.

Ci-dessous vous trouverez un petit échantillon des concepts. C'est particulièrement vrai lorsque l'on considère ce que les Arrays, Strings, etc. peuvent faire. Pour plus de détails vous pouvez regarder ruby-doc (en anglais) ou utiliser votre moteur de recherche favori!

Appeler une méthode est souvent mentionné comme envoyer un message en Ruby. Grosso modo, nous envoyons un message à un objet et nous en attendons une réponse. Selon le message, il peut y avoir zéro, un ou plusieurs arguments. Ainsi nous demandons gentiment à l'objet de faire quelque chose ou de nous donner une information. def nom(paramètre)
  # corps de la méthode
end
 
def salutation(nom)
  puts "Salut " + nom
end

Les méthodes sont des unités de comportement réutilisables. Vous pouvez les définir vous-mêmes comme dans l'exemple. Les méthodes sont petites et focalisées sur l'éxecution d'une tâche spécifique.

La méthode de l'exemple est focalisé sur la salutation de personne. Elle peut être appelé de la manière suivante: salutation("Pierre")

Egalité objet == autre

true == true # => true

3 == 4 # => false

"Bonjour" == "Bonjour" # => true

"Onjour" == "Bonjour" # => false

 Avec deux signes égaux vous pouvez vérifier si deux éléments sont identiques. Si c'est le cas, true sera retourné; sinon, se sera false.
Inégalité objet != autre

true != true # => false

3 != 4 # => true

 L'inégalité est l'inverse d'égalité. C'est à dire qu'il retourne true quand les deux éléments sont différents et false s'ils sont identiques.

Avec if vous pouvez décider que faire selon une condition. Lorsqu'une condition est considérée vraie, alors le code qui suit est éxécuté. Si elle est considérée fausse, alors le code après "else" est éxécuté.

Dans l'exemple, l'accès est accordé si le texte saisi est identique au mot de passe.

 Les constantes ressemblent aux variables, seulement en CAPITALES. Toutes les deux contiennent une valeur et ont un nom pour mentionner ces valeurs. En revanche, alors que la valeur contenu dans une variable peut changer ou être inconnue (si l'on enregistre dans la variable une valeur saisie par l'utilisateur) les constantes sont différentes. Elles ont une valeur connue qui ne devrait jamais changer. Un peu comme les constantes mathématiques ou physiques. Celles-ci ne changent pas, elles font toujours référence à la même valeur.

Arrays

Un array est une collection ordonnée d'éléments indexés par des nombres. Un array contient donc de multiples objets qui sont le plus souvent lié entre eux. Que pouvez-vous donc faire? Vous pouvez regrouper un ensemble de noms de vos fruits favoris dans un array et l'appeler fruits.

C'est seulement une petite sélection de ce qu'un array peut faire. Plus d'informations sur les arrays peuvent être trouvées ici: ruby-doc for Array.

Hashes (Dictionnaires)

Les Hashes associent une clé à une valeur. Vous pouvez ensuite retrouver la valeur par sa clé. Cette construction est appelée dictionnaire dans d'autres languages, ce qui est approprié puisque l'on utilise la clé pour rechercher une valeur, comme vous rechercheriez la définition d'un mot dans un dictionnaire. Chaque clé doit être unique pour un hash donné mais les valeurs peuvent se répéter.

Les hashes peuvent associer n'importe quoi à n'importe quoi! Vous pouvez associer des strings à des nombres, des strings à des strings, des nombres à des booleans... et vous pouvez mélanger tout ces types de valeurs. Cependant les clés sont communément de la même classe. Les symboles sont particulièrement commun comme clés. Les symboles ressemblent à ça: :symbole. Un symbole est une suite de caractères précédées par un deux-points. Vous pouvez les considérer comme des strings spéciaux qui représentent (symbolisent) quelque chose! On utilise souvent les symboles car Ruby éxécute plus rapidement quand on utilise des symboles plutôt que des strings.

Plus d'informations sur les hashes peuvent être trouvées ici: ruby-doc for Hashes.

Concept Utilisation Exemples Description
Créer {clé => valeur}

{:hobbie => "la programmation"} 

{42 => "réponse", "score" => 100,
 :name => "Pierre"}
Vous créez un hash en entourant la paire clé-valeur par des accolades. La flèche va toujours de la clé vers la valeur, ce qui signifie: "La clé pointe vers cette valeur". Les paires clé-valeur sont séparées par des virgules.
Accéder hash[clé] 
hash = {:clé => "valeur"}
hash[:clé] # => "valeur"
hash[foo] # => nil
Accéder à un élément dans un hash ressemble beaucoup à y accéder avec un array. Cependant, avec un hash la clé peut être n'importe quoi, pas uniquement un nombre. Si vous essayez d'accéder à une clé qui n'existe pas, la valeur nil est retournée, ce qui est la manière de dire "rien" de Ruby. Puisqu'il ne reconnait pas la clé il ne peut retourner sa valeur.
Assigner hash[clé] = valeur 
hash = {:a => "b"}
hash[:clé] = "valeur"
hash # => {:a=>"b", :clé=>"valeur"}
Assigner des valeurs à un hash est similaire à l'assignation de valeur à un array. Avec un hash, la clé peut être un nombre ou elle peut être un symbole, une string, un nombre... ou autre !
Supprimer hash.delete(clé) 
hash = {:a => "b", :b => 10}
hash.delete(:a)
hash # => {:b=>10}
Vous pouvez supprimer une clé spécifique d'un hash, de manière que la clé et sa valeur ne peuvent pas être accéder

Bases de Rails

This is an introduction to the basics of Rails. We look at the general structure of a Rails application and the important commands used in the terminal.

The Structure of a Rails app

Here is an overview of all the folders of a new Rails application, outlining the purpose of each folder, and describing the most important files.
Name Description
app This folder contains your application. Therefore it is the most important folder in Ruby on Rails and it is worth digging into its subfolders. See the following rows.
app/assets Assets basically are your front-end stuff. This folder contains images you use on your website, javascripts for all your fancy front-end interaction and stylesheets for all your CSS making your website absolutely beautiful.
app/controllers The controllers sous-dossier contains the controllers, which handle the requests from the users. It is often responsible for a single resource type, such as places, users or attendees. Controllers also tie together the models and the views.
app/helpers Helpers are used to take care of logic that is needed in the views in order to keep the views clean of logic and reuse that logic in multiple views.
app/mailers Functionality to send emails goes here.
app/models The models sous-dossier holds the classes that model the business logic of our application. It is concerned with the things our application is about. Often this is data, that is also saved in the database. Examples here are a Person, or a Place class with all their typical behaviour.
app/views

The views sous-dossier contains the display templates that will be displayed to the user after a successful request. By default they are written in HTML with embedded ruby (.html.erb). The embedded ruby is used to insert data from the application. It is then converted to HTML and sent to the browser of the user. It has subdirectories for every resource of our application, e.g. places, persons. These subdirectories contain the associated view files.

Files starting with an underscore (_) are called partials. Those are parts of a view which are reused in other views. A common example is _form.html.erb which contains the basic form for a given resource. It is used in the new and in the edit view since creating something and editing something looks pretty similar.

config This dossier contains the configuration files that your application will need, including your database configuration (in database.yml) and the particularly important routes.rb which decides how web requests are handled. The routes.rb file matches a given URL with the controller which will handle the request.
db Contains a lot of database related files. Most importantly the migrations sous-dossier, containing all your database migration files. Migrations set up your database structure, including the attributes of your models. With migrations you can add new attributes to existing models or create new models. So you could add the favorite_color attribute to your Person model so everyone can specify their favorite color.
doc Contains the documentation you create for your application. Not too important when starting out.
lib Short for library. Contains code you've developed that is used in your application and may be used elsewhere. For example, this might be code used to get specific information from Facebook.
log See all the funny stuff that is written in the console where you started the Rails server? It is written to your development.log. Logs contain runtime information of your application. If an error happens, it will be recorded here.
public Contains static files that do not contain Ruby code, such as error pages.
script By default contains what is executed when you type in the rails command. Seldom of importance to beginners.
test Contains the tests for your application. With tests you make sure that your application actually does what you think it does. This dossier might also be called spec, if you are using the RSpec gem (an alternative testing framework).
vendor A folder for software code provided by others ("libraries"). Most often, libraries are provided as ruby gems and installed using the Gemfile. If code is not available as a ruby gem then you should put it here. This might be the case for jQuery plugins. Probably won't be used that often in the beginning.
Gemfile

A file that specifies a list of gems that are required to run your application. Rails itself is a gem you will find listed in the Gemfile. Ruby gems are self-contained packages of code, more generally called libraries, that add functionality or features to your application.

If you want to add a new gem to your application, add "gem gem_name" to your Gemfile, optionally specifying a version number. Save the file and then run bundle install to install the gem.

Gemfile.lock This file specifies the exact versions of all gems you use. Because some gems depend on other gems, Ruby will install all you need automatically. The file also contains specific version numbers. It can be used to make sure that everyone within a team is working with the same versions of gems. The file is auto-generated. Do not edit this file.

Important Rails commands

Here is a summary of important commands that can be used as you develop your Ruby on Rails app. You must be in the root dossier of your project to run any of these commands (with the exception of the rails new command). The project or application root dossier is the folder containing all the subfolders described above (app, config, etc.).

Concept Usage Description
Create a new app rails new name Create a new Ruby on Rails application with the given name here. This will give you the basic structure to immediately get started. After this command has successfully run your application is in a folder with the same name you gave the application. You have to cd into that folder.
Start the server rails server

You have to start the server in order for your application to respond to your requests. Starting the server might take some time. When it is done, you can access your application under localhost:3000 in the browser of your choice.

In order to stop the server, go to the console where it is running and press Ctrl + C
Scaffolding rails generate scaffold name attribute:type

The scaffold command magically generates all the common things needed for a new resource for you! This includes controllers, models and views. It also creates the following basic actions: create a new resource, edit a resource, show a resource, and delete a resource.

That's all the basics you need. Take this example:

rails generate scaffold product name:string price:integer

Now you can create new products, edit them, view them and delete them if you don't need them anymore. Nothing stops you from creating a full fledged web shop now ;-)
Run migrations rake db:migrate When you add a new migration, for example by creating a new scaffold, the migration has to be applied to your database. The command is used to update your database.
Install dependencies bundle install If you just added a new gem to your Gemfile you should run bundle install to install it. Don't forget to restart your server afterwards!

Help: What to do when things go wrong?

Things go wrong all the time. Don't worry, this happens to everyone. So keep calm. When you encounter an error, just google the error message. For best results, add the keywords "rails" or "ruby". Results from stackoverflow.com are often really helpful. Look for those! The most experienced developers do this frequently ;-).

Here are common mistakes with a little checklist:

  • Have you run rake db:migrate to apply the newest database migrations?
  • Have you really saved the file you just changed? Unsaved files are often marked in the editor via an asterisk or a point next to their name.
  • If you just added a gem to the Gemfile, have you run bundle install to install it?
  • If you just installed a gem, have you restarted the server?
